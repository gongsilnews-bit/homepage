# 백엔드 JSON 하위 호환성 유지 전략 (Frontend 무중단 마이그레이션)

이 문서는 기존의 RDBMS 일반 컬럼들을 JSONB 컬럼 하나(`settings`)로 통폐합하는 대규모 DB 마이그레이션 과정에서, **기존에 잘 작동하던 프론트엔드(React/어드민) 코드를 단 한 줄도 수정하지 않고도 시스템이 완벽하게 돌아가게 만드는 백엔드(Server Actions) 중간 번역(Adapter) 기법**을 정리한 것입니다.

---

## 1. 개요 (문제 상황)
기존 DB 구조에서는 `logo_url`, `company_intro` 등이 각각의 독립된 컬럼으로 존재했습니다. 이 상태에서 어드민 프론트엔드는 폼 데이터를 주고받을 때 `data.logo_url` 같은 평탄화(Flat)된 구조를 사용하고 있었습니다.

만약 DB 컬럼들을 지우고 `settings` JSON 안쪽의 `settings -> header -> logo_url` 구조로 변경해버리면, 기존의 모든 프론트엔드 코드가 에러를 뿜으며 화면이 멈추는(Breaking Change) 단점이 발생합니다.

## 2. 해결 방법: Server Actions 중간 번역(Adapter) 채택

이 문제를 해결하기 위해 프론트엔드를 고치는 대신, 백엔드인 Next.js의 **Server Actions (`src/app/actions/homepage.ts`)** 에 번역기 역할을 부여했습니다.

### A. 가져올 때 (Read / `getHomepageSettings`)
- **과정:** DB에서는 JSONB(`settings`) 객체 하나를 통째로 가져옵니다.
- **번역:** 백엔드 함수 내부에서 이 객체를 해체하여, 기존 프론트엔드가 알고 있는 플랫(Flat)한 스펠링으로 평탄화 작업을 거칩니다.
- **결과:** 프론트엔드는 DB가 마이그레이션된 사실조차 모른 채 기존과 완벽하게 동일한 구조(`{ logo_url, company_intro, ... }`)를 리턴받습니다.

### B. 저장할 때 (Write / `saveHomepageSettings`)
- **과정:** 프론트엔드는 기존 방식 그대로 평탄화된 데이터를 서버로 보냅니다.
- **번역:** 
  1. 백엔드는 DB에 저장된 최신 JSON 데이터를 먼저 조회(`Deep Merge`를 위함)합니다.
  2. 프론트엔드가 보내준 조각 데이터들을 깊은 종속성을 가진 JSON 구조(`settings.header.logo_url` 등)의 제자리에 알맞게 밀어 넣습니다.
- **결과:** 어드민 페이지에서 '저장'을 누르면 DB에는 깔끔하게 구조화된 JSON 데이터로 안전하게 덮어쓰기 저장(Upsert)됩니다.

## 3. 이 아키텍처의 비즈니스적 장점
1. **무중단 배포 (Zero Downtime):** DB 구조를 완전히 갈아엎는 메이저 업데이트임에도 불구하고 프론트엔드쪽 버그나 서비스 중단이 아예 발생하지 않았습니다.
2. **프론트엔드 개발자 리소스 절약:** 복잡한 어드민 State 관리 코드를 손댈 필요가 없어 작업 시간이 단축되었습니다.
3. **유연한 미래 전환:** 당장은 하위 호환성을 유지하다가, 나중에 시간이 날 때 프론트엔드를 JSON 구조에 맞추어(React Hook Form 재설계 등) 여유롭게 리팩토링할 수 있는 시간적 벌이를 해줍니다.
